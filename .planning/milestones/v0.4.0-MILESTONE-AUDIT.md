## Cross-Phase Integration (4/4 E2E flows)

1. **Capture → stamp → edit:** A capture made while a mode is active is stamped (Phase 18), and the
   stamp is editable/correctable after the fact via the Phase 17 "Other metadata" editor — the
   designed 18↔17 link holds. ✅
2. **First-run activation → dashboard → stamp:** DashboardPage idempotently activates the first
   layout as the default mode on first run, so the dashboard is never empty, the app bar shows state,
   and captures are stamped from day one. ✅
3. **Menu switch → app bar → dashboard:** Activating a mode from the AppShell menu updates the app
   bar and changes which mode's buttons the dashboard shows. ✅
4. **Date default → editable:** `occurredAt` defaults to today on capture (Phase 16) and a wrong
   default is trivially fixable via the Phase 17 edit form (the designed 16↔17 pairing). ✅

## North-Star Check (seeds/fewest-buttons-slickest)

The milestone REMOVES steady-state controls rather than adding them: the ever-present on-dashboard
layout switcher and non-active modes are gone; switching moved to the rarely-used hamburger menu;
the date field is defaulted rather than always-filled. Net on-screen control count is lower. ✅

## Tech Debt / Notes (non-blocking)

- `activeLayoutRepository` / `useActiveLayoutName` are now unused by the dashboard (still used by
  `ManageShortcutsPage` for layout selection). Harmless duplication; the active MODE is the dashboard's
  source of truth. Candidate for a future cleanup that unifies "layout being edited" vs "active mode".
- If a layout backing the active mode is renamed/deleted in the authoring tool, the active-mode name
  can go stale; the dashboard falls back to `layouts[0]` gracefully. Acceptable for a personal prototype;
  a future enhancement could re-point or re-prompt the active mode on layout rename.
- A capture made before ever visiting the dashboard (no mode yet activated) is not stamped — consistent
  with STAMP-01 ("no stamp when inactive"); the first dashboard visit activates a default mode.
