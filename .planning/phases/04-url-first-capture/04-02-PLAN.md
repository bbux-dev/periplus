  <implementation>
    RED first: write extractMetadataFromUrl.test.ts encoding all 16 fixtures as `it`/`expect` cases (group by domain with describe blocks). Run — it fails (no impl). GREEN: implement the function. REFACTOR: extract per-domain helpers + a shared slug-titlecase helper.

    Dispatch (RESEARCH "Dispatch Strategy"): start with `const base: ExtractedDraft = { sourceUrl: url, metadata: {} }`. Wrap `new URL(url)` in try/catch; on catch return base (handles "not-a-url" — CAPT-04). Then branch on `parsed.hostname`:
      - hostname === 'maps.app.goo.gl' → return { sourceUrl: url, metadata: { extractionNote: 'shortened-link-unresolvable-offline' } } (no title/location).
      - hostname includes 'google.com' AND pathname starts with '/maps' → Google Maps extractor.
      - hostname includes 'imdb.com' → IMDb extractor.
      - hostname includes 'goodreads.com' → Goodreads extractor.
      - hostname includes 'amazon.' → Amazon extractor.
      - hostname includes 'podcasts.apple.com' → Apple Podcasts extractor.
      - hostname === 'open.spotify.com' → Spotify extractor.
      - else → return base (unrecognized host, e.g. example.com — CAPT-04).

    Google Maps: place name = pathname match /\/maps\/place\/([^/@]+)/, decoded via a `decodeGoogleMapsName` helper that does `decodeURIComponent(raw.replace(/\+/g, '%20'))` inside a try/catch (fallback `raw.replace(/\+/g, ' ')`). This is the CRITICAL `+`→space pitfall: the WHATWG URL API does NOT decode `+` in pathnames. Set BOTH title and location to the place name. Coordinates: match /@(-?\d+\.\d+),(-?\d+\.\d+)/ on pathname; fallback to /!3d(-?\d+\.\d+)!4d(-?\d+\.\d+)/ on searchParams.get('data'); when both lat & lng found, set metadata.coordinates = { lat: parseFloat(lat), lng: parseFloat(lng) }, else metadata {}.

    IMDb: match /\/title\/(tt\d+)/ on pathname → metadata.imdbId when matched, else metadata {} (the /name/ person URL yields no imdbId). title is never set (not in URL).

    Goodreads: match /\/book\/show\/\d+-(.+)/ on pathname → slug → titlecase helper.

    Amazon: find '/dp/' in pathname; if absent return base. Take the last non-empty path segment BEFORE '/dp/'; if present, decodeURIComponent then titlecase to a title; else title undefined. IMPORTANT — the fixture for "Pragmatic-Programmer-journey-mastery-Anniversary" expects "Pragmatic Programmer Journey Mastery Anniversary" (TITLE-CASED). The RESEARCH code sketch omitted titlecasing; the FIXTURE is authoritative, so apply the SAME titlecase helper used for Goodreads/Apple.

    Apple Podcasts: match /\/podcast\/([^/]+)\/id\d+/ on pathname → slug → titlecase helper; metadata {}.

    Spotify: match /\/(show|episode)\/([^/]+)/ on pathname → metadata { spotifyId, kind }; title never set.

    Shared titlecase helper: replace(/-/g, ' ') then capitalize the first letter of each word — `.replace(/\b\w/g, c => c.toUpperCase())`. Verify against fixtures: "the-pragmatic-programmer" → "The Pragmatic Programmer"; "a-brief-history-of-time" → "A Brief History Of Time"; "the-daily" → "The Daily".

    Export both `ExtractedDraft` (interface) and `extractMetadataFromUrl`. Import `EntryType` as a type-only import from './db'. The `type` parameter is part of the signature (CAPT-03) and reserved for type-based fallback dispatch; hostname dispatch covers all 16 fixtures, so type-based fallback is currently a no-op returning base — keep the parameter, do not remove it.
